## 1. Purpose and scope

This Software Safety Plan (SSP) defines the lifecycle, activities, methods, and
responsibilities for the development and verification of cpp-CAN
(`github.com/SoundMatt/cpp-CAN`) in accordance with:

- ISO 26262:2018 — Road vehicles — Functional Safety (Parts 3, 4, 6, 8)
- IEC 61508:2010 — Functional Safety of E/E/PE Safety-related Systems (Part 3)

cpp-CAN is developed as a **Safety Element Out Of Context (SEOOC)** targeting
ASIL-B (ISO 26262) / SIL 2 (IEC 61508). Refer to `HARA.md` for the hazard
analysis and risk assessment that derives these levels.

**Out of scope:** System-level HARA, hardware fault model (FMEDA), airworthiness
(DO-178C), AUTOSAR integration. These are the integrating system's responsibility.

## 4. Development lifecycle

### 4.1 Branching strategy

All changes follow the `feat/<name>` → PR → `main` workflow.  
`main` is protected: every PR requires passing CI and CODEOWNERS approval.

### 4.2 Requirements management

Requirements are maintained in `requirements/requirements.json` using the
SoundMatt REQ-MODULE-NNN numbering scheme. Every requirement must be:

1. Implemented with a `// fusa:req REQ-MODULE-NNN` annotation
2. Tested with a `// fusa:test REQ-MODULE-NNN` annotation
3. Verified green by `cpfusa trace` in CI

### 4.3 Verification

| Activity | Tool | Gate |
|---|---|---|
| Unit tests | Catch2 | 100% pass on all 5 platforms |
| Static analysis | clang-tidy | 0 errors |
| ASIL-B qualification | cpfusa qualify | must pass |
| ISO 26262 gap analysis | cpfusa iso26262 --asil ASIL-B | informational |
| IEC 61508 gap analysis | cpfusa iec61508 --sil SIL-2 | informational |
| Vulnerability scan | cpfusa vuln | informational |
| Coverage | LCOV | informational |

- SBOMs are generated by `cpfusa release` and stored as `sbom.json`

## 6. Known limitations and assumptions of use (ASIL-B SEOOC)

1. The virtual bus is intended for in-process testing only; it provides no
   hardware isolation or fault containment.
2. ISO-TP and J1939 implementations assume correct underlying CAN hardware;
   they do not implement hardware-level error counting or bus-off recovery.
3. The E2E protection (CRC-16/CCITT-FALSE) provides detection, not correction;
   the integrating system must handle rejected frames.
4. Thread safety is provided by `std::shared_mutex` and `std::atomic`; the
   integrating RTOS must support these primitives.
